# 一、智能管理系统项目介绍

## 1.1 项目梗概
这是一个加入了基于MCP的大模型的管理系统demo，管理系统基于vue+springboot，并加入了Redis和消息队列以接近于真实的开发需求

## 1.2项目文件介绍
这个项目由两个大文件夹，前端（xx_managersystem）和后端（XX_mangaer_system）构成
前端基于Vue3构建，主要包含登录（login.vure）和主页面（index.vue），是主要的图形化操作页面。
后端基于springboot构造，并引入mybatis-plus和Redis缓存池（demo阶段未启用Redis），负责主要的业务逻辑和数据库处理以及集成大模型进行智能化。

## 二、核心技术栈
| 技术类别       | 选型及能力说明                                                                 |
|----------------|------------------------------------------------------------------------------|
| 基础框架       | Spring Boot<br>- 作为核心开发框架，提供依赖注入、Web 接口、配置管理等基础能力   |
| 协议集成       | MCP（Model Context Protocol）协议<br>- 标准化大模型与后端工具/服务的交互       |
| 大模型引擎     | DeepSeek R1<br>- 负责自然语言理解、口语化请求转标准指令、生成口语化响应         |
| 存储技术       | Redis + 内存双模式<br>- 用于用户对话上下文的持久化与缓存                       |

## 三、智能管理系统实现要点
### 3.1 用户交互层
- 提供 `/mcp/chat` 统一接口接收用户口语化请求
- 支持「直接指令格式」和「口语化请求」两种输入形式，区分处理逻辑
- 最终返回口语化响应，适配自然交互体验

### 3.2 大模型交互层
- 上下文构建：封装系统提示 + 用户输入，生成符合 DeepSeek R1 调用规范的消息上下文
- 指令转换：调用 DeepSeek R1 将口语化请求转化为标准化指令（如 `/add-user username=xxx password=xxx`）
- 响应处理：解析模型返回结果，提取有效指令内容

### 3.3 指令处理层
- 指令解析：校验指令格式有效性，提取操作类型（如 add-user）和参数（username/password）
- 指令执行：映射指令到后端 CRUD 任务（用户管理为主），调用对应业务服务
- 结果封装：将任务执行结果转换为自然语言响应

### 3.4 上下文管理层
- 多存储策略：支持 Redis（分布式场景）和内存（单机/测试场景）两种上下文存储方式
- 会话一致性：维护用户对话上下文，保障多轮交互的连续性
- 异常兼容：处理 Redis 连接失败等存储异常，兜底保障系统可用

## 四、核心组件设计要点
- 分层解耦：按「接口层（McpController）- 上下文层（McpContextService）- 模型层（McpModelService）- 解析层（McpCommandParserService）- 业务层（McpUserService）」分层设计，职责单一且可扩展
- 标准化交互：基于 MCP 协议规范大模型与后端服务的通信，降低耦合
- 异常鲁棒性：针对 Redis 连接、模型调用、指令解析等关键环节设计异常处理机制，保障系统稳定性
- 可配置化：模型调用参数、上下文存储方式等支持配置化管理，适配不同部署场景

## 五、核心技术特性
- 自然语言到机器指令的转换：核心依赖 DeepSeek R1 的语义理解能力，实现非结构化口语请求到结构化指令的转化
- 上下文感知：通过上下文存储机制，让智能体具备多轮对话理解能力
- 业务解耦：指令解析层与业务服务层分离，新增业务操作仅需扩展指令映射和业务服务，无需修改核心流程
- 用户体验优化：输入输出均为自然语言，降低用户使用门槛

## 六、项目须知
- 由于接口限制，智能体在查询信息时只能通过ID字段查询
- 项目尚处于demo阶段，很多功能并不完善
- 项目旨在探索未来高度AI化，自动化的后端管理系统，但是一方面MCP协议也刚刚成熟，许多技术路线还有待发展；另一方面，MCP协议面临安全成本大，技术风险高，词元消耗快的问题，这导致项目落地还需要很长时间
